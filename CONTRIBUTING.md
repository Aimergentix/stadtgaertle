# Contributing to Stadtgaertle Rheinfelden

This project uses the **HACE** (Human — AI Constraint Engine) methodology for AI-assisted development. Every contribution — whether from the primary Archon ('user', 'human pair programmer') or a future collaborator — follows the same structured workflow described below.

---

## Start here

If you are new to this project, your entry point is one file:

**[`docs/ARCHON_START_HERE.md`](docs/ARCHON_START_HERE.md)**

Read it fully before touching any code or opening any chat session.
It explains the tools, the workflow, and the habits that keep this project clean.

---

## How this project is built

This is an **Architecture-Driven, AI-Assisted** project. Code is not written freely —
it is generated by an AI pair-programmer (the **Aimergent**) inside tightly scoped
sessions called **Small Worlds**, reviewed line by line by the human owner/pair-programmer (the
**Archon**), and only then committed to the repository.

| Role      | Definition                                                           |
|-----------|----------------------------------------------------------------------|
| Archon    | The human owner. Designs blueprints, reviews all output, holds final commit gate. |
| Aimergent | The AI session (any model). Generates exactly one file per atomic request. Never writes autonomously. |

The full methodology is defined in [`HACE.md`](HACE.md).
The structural law for the AI session is in [`.cursor/rules/hace.mdc`](.cursor/rules/hace.mdc).

---

## Before making any change

1. Read the relevant document in [`docs/`](docs/) before opening a coding session.
2. Open a **new, empty** Cursor Side-Chat for each task. Never reuse an old chat.
3. Use only the `@`-mentions listed in [`prompts/layer-prompts.md`](prompts/layer-prompts.md).
4. Output is reviewed in chat first. Files are created manually after review.
5. One Small World = one file = one commit.

---

## Code standards

All rules are enforced mechanically by ESLint and documented in `.cursor/rules/hace.mdc`.
The short version:

- No `any` in TypeScript — use `unknown` plus type guards or an explicit interface.
- No `// TODO`, `// FIXME`, or partial implementations.
- No hardcoded outbound URLs — they live only in `frontend/src/config/services-router.ts`.
- No visible copy inlined in components — it lives only in `frontend/src/config/site-content.json`.
- Hard cap: 200 lines per file. Target: under 100.
- One React component per file. File name = component name.

## Commit conventions

```
feat: add CalendarSection component       ← new Small World complete
fix: correct type error in SiteContent    ← repair within a Small World
chore: scaffold layer-1-foundation        ← tooling / setup work
docs: update 03_assembly_line phase-2     ← documentation changes
```

One Small World per commit. The commit message names the assembly-line step.

---

## Proposing a change to the specification

If you believe a structural decision (tech stack, directory layout, design system) needs
to change, open a **GitHub Issue** before writing any code. Label it `spec-change`.
Describe: what you want to change, which doc section it affects, and why.

No code that conflicts with [`docs/01_website_spec.md`](docs/01_website_spec.md) or
[`HACE.md`](HACE.md) will be merged without a prior Issue and an update to the
relevant document.
